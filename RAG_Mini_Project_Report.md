## Abstract

Retrieval-Augmented Generation, commonly called RAG, is a practical technique used to improve the quality of answers generated by large language models. Instead of asking a language model to answer only from its pre-trained knowledge, RAG first searches relevant information from a custom knowledge base and then provides that information to the model as context. This makes the final answer more relevant, grounded, and useful for domain-specific tasks.

This mini project presents the design and implementation of a simple but functional RAG pipeline for answering questions from local files stored on a laptop. The system accepts user documents such as PDF, TXT, CSV, Excel, DOCX, and JSON files from a local data folder. It reads and loads the content, splits it into smaller chunks, converts each chunk into a vector embedding using a sentence transformer model, stores those vectors inside a FAISS index, and performs similarity search when the user asks a query. The retrieved text chunks are then passed to a large language model through the Groq API to generate a natural language answer.

The implementation has been completed in Python using libraries such as LangChain Community loaders, Sentence Transformers, FAISS, python-dotenv, and langchain-groq. The project demonstrates the complete RAG workflow including data ingestion, text chunking, embedding generation, vector storage, semantic retrieval, and answer generation. It also shows how a project based on modern AI concepts can be built on an ordinary student laptop.

This project is especially useful as a college mini project because it combines theoretical AI concepts with practical implementation. It introduces the learner to embeddings, vector databases, retrieval, prompt context, and LLM-based generation. The final system can answer questions from uploaded documents and therefore serves as a working example of how knowledge-grounded AI applications are created.

## 1. Introduction

Artificial Intelligence has experienced rapid growth in recent years, especially in the area of natural language processing. Large language models can summarize text, answer questions, rewrite content, generate code, and assist users in a wide range of tasks. However, a major limitation of general-purpose language models is that they do not automatically know the user’s private or local documents. They generate answers mainly from patterns learned during pre-training. As a result, their responses can become generic, outdated, or disconnected from the exact content that a user wants to query.

Retrieval-Augmented Generation solves this problem by combining two important ideas. The first idea is **retrieval**, which means searching a custom knowledge base and selecting the most relevant pieces of text for a given question. The second idea is **generation**, which means using a language model to produce a fluent final answer. When these two steps are combined, the language model receives better context and can answer more accurately from the user’s own data.

This project demonstrates a traditional RAG system in a simple local setup. The user places files inside a `data` folder. The program loads those files, extracts text, splits it into chunks, converts the chunks into numerical vectors called embeddings, and stores them in a FAISS vector store. When the user types a question, the system converts that question into an embedding, searches the vector store for similar chunks, and sends the retrieved context to a large language model. The model then writes the answer using the retrieved information.

The project is useful from both academic and practical perspectives. Academically, it introduces key concepts such as embeddings, similarity search, vector stores, chunking, retrieval, and prompt augmentation. Practically, it creates a working application that answers questions from documents such as reports, notes, and study material.

## 2. Problem Statement

Large language models are powerful, but they cannot reliably answer questions from private local files unless those files are given as context. If a user uploads a PDF report and asks questions about that report, a normal LLM may guess based on general knowledge instead of using the real document content. This leads to low trust, hallucinated answers, and poor domain relevance.

The problem addressed in this project is:

**To design and implement a document-based Retrieval-Augmented Generation system that can read local files, build a searchable vector representation of their contents, retrieve context relevant to a user query, and generate a final answer grounded in the uploaded documents.**

This problem includes several smaller technical challenges:

- multiple file formats must be supported
- raw text must be loaded correctly from user documents
- long text must be split into manageable chunks
- semantic vector embeddings must be generated efficiently
- those vectors must be indexed for fast similarity search
- the most relevant chunks must be retrieved at query time
- an LLM must use those chunks to generate a meaningful final answer
- the entire workflow should run on a student laptop using accessible tools

## 4. Motivation and Need for RAG

The motivation behind RAG comes from the gap between general AI knowledge and specific user knowledge. A language model such as a chatbot can answer many general questions, but it does not automatically know the contents of a student’s report, institute notes, assignment PDF, or private study material. If we ask such a model directly, it may still produce a polished answer, but the answer may not come from the actual file. This is risky in academic, business, legal, healthcare, and enterprise settings.

RAG is needed because it helps the model answer from the right source. It is cheaper than fine-tuning for many use cases and much easier to update. If the data changes, we do not retrain the model. We simply update the documents and rebuild the vector store. This makes RAG an efficient and scalable solution.

Some practical reasons for using RAG are listed below:

- it improves answer relevance
- it helps reduce hallucination
- it allows private document question answering
- it supports dynamic knowledge updates
- it avoids costly model retraining in many cases
- it is suitable for organization-specific or subject-specific information

## 6. Background Concepts

### 6.1 What is a Large Language Model?

A large language model is a deep learning model trained on vast amounts of text. It learns patterns in language and can generate human-like responses. However, its knowledge is limited to the data seen during training and it may not know the user’s latest or private documents.

### 6.2 What is Retrieval?

Retrieval means searching stored information and finding the pieces that are most relevant to a question. In a RAG system, retrieval is usually based on semantic similarity, not exact keyword matching only.

### 6.3 What is Generation?

Generation means producing a final response using a language model. In RAG, the generation step happens after retrieval so that the model can answer using supplied context.

### 6.4 What is an Embedding?

An embedding is a dense numerical vector representation of text. Similar pieces of text tend to have similar vectors. Embeddings make semantic search possible.

### 6.5 What is a Vector Store?

A vector store is a database-like structure used to save embeddings and perform similarity search. In this project, FAISS is used as the vector store.

### 6.6 What is Chunking?

Chunking means dividing a long document into smaller pieces. This is necessary because embeddings and retrieval work better on manageable text segments than on entire long documents.

### 6.7 What is Similarity Search?

Similarity search compares the query vector with stored document vectors and returns the closest matches. These matches are assumed to be the most relevant text chunks for answering the query.

## 8. Existing Approach vs Proposed Approach

### Existing Direct LLM Approach

In a direct LLM approach, the user asks a question and the model answers from its training knowledge. This is fast and simple, but it has major drawbacks:

- it may not know the user’s documents
- answers may be generic
- it may hallucinate facts
- it cannot easily adapt to local private files

### Proposed RAG Approach

In the proposed approach, the system first searches the user’s uploaded files and finds relevant chunks before asking the LLM to answer. This improves grounding and domain relevance.

Advantages of the proposed approach include:

- answers come from user data
- documents can be updated without retraining
- domain-specific question answering becomes possible
- the model receives context before responding

## 11. Project Workflow

The project follows a clear pipeline:

1. User places documents in `data` folder.
2. Program loads files using appropriate document loaders.
3. Loaded text is converted into document objects.
4. Documents are split into smaller chunks.
5. Each chunk is converted to an embedding vector.
6. Embeddings are stored in a FAISS index with metadata.
7. User types a question in the terminal.
8. Query is converted into an embedding.
9. Similarity search retrieves the most relevant chunks.
10. Retrieved chunks are joined as context.
11. Context plus question are sent to Groq LLM.
12. LLM returns a final natural-language answer.

This is the complete traditional RAG loop implemented in the project.

---

## 12. Data Ingestion Pipeline

Data ingestion is the first major step of the system. The purpose of this step is to read files from the `data` directory and convert them into a standard internal format that the rest of the pipeline can process.

In this project, the file `src/data_loader.py` performs data ingestion. It scans the folder for supported file extensions and uses specific loaders for each type. For example, PDFs are handled by `PyPDFLoader`, text files by `TextLoader`, CSV files by `CSVLoader`, DOCX files by `Docx2txtLoader`, Excel files by `UnstructuredExcelLoader`, and JSON files by `JSONLoader`.

The output of this step is a list of document objects containing text and metadata. This standardization is important because the downstream chunking and embedding code should not care whether the original source was a PDF, TXT, or CSV file.

---

## 13. Text Chunking Strategy

After loading the documents, the next step is chunking. Long text is hard to search effectively if kept as one huge block. Small meaningful chunks improve retrieval quality. In this project, chunking is performed using `RecursiveCharacterTextSplitter`.

The chunking configuration used in the project is:

- chunk size: 1000 characters
- chunk overlap: 200 characters

Overlap is included so that important information near chunk boundaries is not lost completely. For example, if one sentence starts at the end of one chunk and continues into the next, overlap helps preserve continuity.

Chunking matters because the vector store works on chunk-level embeddings. If chunks are too large, retrieval becomes noisy. If chunks are too small, context becomes fragmented. The chosen values provide a reasonable balance for a college-level implementation.

---

## 14. Embeddings and Vector Representation

The embedding step converts text into numbers that capture semantic meaning. In this project, the model used is `all-MiniLM-L6-v2`, a widely used sentence transformer model that is lightweight and suitable for local use.

Each text chunk becomes a 384-dimensional numerical vector. These vectors are not manually designed features. They are learned representations generated by the pre-trained transformer model. Similar pieces of text tend to have vectors that are closer together in vector space.

For example, text about "model selection" and text about "choosing YOLOv8" may be semantically closer than unrelated text about references or certificates. This semantic property is what makes vector-based retrieval powerful.

---

## 15. Vector Store Design Using FAISS

FAISS stands for Facebook AI Similarity Search. It is a library designed for efficient similarity search over dense vectors. In this project, FAISS is used as a local vector store to hold the embeddings of all document chunks.

The vector store is created in `src/vectorstore.py`. A `faiss.IndexFlatL2` index is used, which performs exact search based on L2 distance. When embeddings are added, the code also stores metadata for each chunk so that the retrieved vectors can be mapped back to the original text.

The vector store is persisted locally in the folder `faiss_store` using two files:

- `faiss.index` for the actual vector index
- `metadata.pkl` for corresponding text metadata

This persistence is useful because the index does not need to be rebuilt every time unless the input files change.

---

## 16. Retrieval Pipeline

When the user asks a question, the system does not immediately call the LLM. First, it performs retrieval. The query is converted into an embedding using the same sentence transformer model used for document chunks. This keeps both document vectors and query vectors in the same semantic space.

The vector store then compares the query embedding with stored chunk embeddings and returns the top `k` nearest chunks. In this project, `top_k=3` is used by default. The text of these retrieved chunks becomes the context for the LLM.

This retrieval step is the most important difference between normal LLM usage and RAG. It ensures that the model sees source information from the user’s files before generating the answer.

---

## 17. LLM Integration with Groq

The project uses Groq through `langchain_groq`. Groq provides very fast inference for supported open models. In the updated project, the model used is `llama-3.1-8b-instant`.

The API key is stored in the `.env` file under `GROQ_API_KEY`. The file `src/search.py` loads this key and creates the LLM client. Once the top relevant chunks are retrieved, the system creates a prompt containing:

- the user query
- the retrieved context
- an instruction to summarize or answer based on that context

The LLM then generates the final answer in natural language.

---

## 18. Final Architecture of the Project

The architecture of the project can be summarized as follows:

**Input Files -> Document Loaders -> Chunking -> Embeddings -> FAISS Vector Store -> Query Embedding -> Similarity Search -> Retrieved Context -> Groq LLM -> Final Answer**

This architecture is known as traditional RAG because retrieval is performed first and the LLM response is augmented by the retrieved context.

## 19. Implementation Details

The implementation is modular and divided into source files so that each stage of the pipeline has a clear responsibility.

### `app.py`

This is the main entry point of the project. It checks whether the FAISS index already exists. If not, it loads documents and builds the vector store. Then it creates a `RAGSearch` object and repeatedly asks the user for questions in the terminal until the user types `exit`.

### `src/data_loader.py`

This file loads documents from the `data` directory using the correct loader for each supported format. It returns a list of document objects.

### `src/embedding.py`

This file handles chunking and embedding generation. It loads the sentence transformer model and exposes methods for splitting documents and embedding their chunks.

### `src/vectorstore.py`

This file builds, saves, loads, and queries the FAISS vector store. It stores embeddings along with chunk text metadata.

### `src/search.py`

This file combines retrieval and generation. It ensures the vector store is available, retrieves top chunks for the user query, builds a context string, and sends that context to the Groq LLM.
